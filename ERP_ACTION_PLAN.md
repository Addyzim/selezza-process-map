## Фаза 2 — конструктор ролей (RBAC)

Замечание: «нужно добавить возможность создавать роли в админке и контролировать,
какие компоненты можно контролировать в зависимости от роли (галочки)».

### [x] T2a. Модель данных ролей и права-флаги — СДЕЛАНО (2026-07-02, Sonnet)
- **Единый список прав:** новый `app/core/permissions.py` — `PERMISSIONS` (код + ключ локали
  для UI), 9 флагов по фактическим разделам консоли (1:1 с пунктами сайдбара из T4a):
  `perm_board, perm_orders, perm_catalog, perm_stock, perm_team, perm_analytics, perm_messages,
  perm_admin_users, perm_admin_roles`.
- **Осознанное решение:** `is_management` и `can_sew` НЕ включены в этот список и не тронуты —
  это тип аккаунта (доступ к консоли управления вообще / доступ к мобильному виду швеи), а не
  покомпонентное право; их удаление задело бы вход/маршрутизацию гораздо шире, чем нужно для
  замечания шефа. `board_view_all`/`board_sew`/`stock_view`+`stock_edit` из черновика плана не
  реализовывались буквально — в коде нет разделения «просмотр vs правка» внутри модуля (все
  экшены одного роутера и так шли одним доступом), так что права взяты по модулю (1 флаг = 1
  роутер), а не по действию — не выдумывал более тонкую грануляцию, которой в коде не было.
- **Модель:** `Role` (`app/modules/users/models.py`) получила 9 `perm_*`-колонок + `is_builtin`
  (встроенную роль нельзя удалить). Схема синхронизирована в `db/schema.sql`
  (структура + `INSERT INTO roles`), и в `app/seed.py` (SQLite dev).
- **Ничего не сломано:** все 4 встроенные роли (ГД/Владелец/Гл.цеха/Швея) получили ВСЕ `perm_*`
  = TRUE при сиде — раньше `is_management=True` уже давал полный доступ ко всей консоли, миграция
  не должна ничего отобрать у существующих пользователей. Ограниченные права — только у новых
  ролей, создаваемых через конструктор.
- **Готово, когда:** роли хранятся как данные (были и раньше), встроенные роли посеяны с полными
  правами, старые пользователи работают как раньше — подтверждено вживую (см. T2c).

## Фаза 3 — модель статусов: «на месте» ≠ «доступна» ≠ «занята»

Замечание: «Занятость и доступность не всегда 1в1 идут с "на месте". Человек может быть
на месте, но быть недоступной. Если шеф сам делает статус швеи — занята — то у швеи это
должно показываться как какая-то задача, которая её занимает».

### [x] T3a. Разделить статус на три независимых измерения — СДЕЛАНО (2026-07-02, Sonnet)
- **Модель:** `User` теперь хранит `is_present` (новое поле, чек-ин швеи в мобильном виде) и
  `is_available` (решение управления). `is_occupied` НЕ хранится — вычисляется в новом
  `app/modules/team/service.py` (`is_occupied()` / `occupied_map()`) по наличию карточки
  «в работе»; TODO-комментарий отмечает, где T3b добавит проверку ручной задачи через OR.
  Схема синхронизирована: `db/schema.sql` (users) получил оба столбца — раньше `is_available`
  там вообще отсутствовал (drift между ORM и schema.sql), теперь исправлено. Alembic/migrations
  в репо нет (только `db/schema.sql` + `Base.metadata.create_all` для dev/SQLite) — новых
  миграций заводить было не из чего, схема просто обновлена как источник правды.
- **Роутеры:** `mobile/router.py` — `/m/availability` переименован в `/m/checkin`, переключает
  `is_present` (было спутано с `is_available`). `team/router.py` — переключатель по-прежнему

  показывается предупреждение «Управление отметило вас недоступной» (не блокирует работу —

## Правила для сессий Sonnet
- Одна задача за сессию; не расползаться на соседние задачи «раз уж рядом».
